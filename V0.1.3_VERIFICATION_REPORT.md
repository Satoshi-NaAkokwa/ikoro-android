## Root cause of the wallet crash

`WalletScreen` was calling `ServiceLocator.walletManager(context)` inside `remember { ... }` during Compose composition (main thread). `WalletManager` then eagerly called `ServiceLocator.identityManager(context)`, which constructs `IdentityStore`. `IdentityStore` creates `EncryptedSharedPreferences` and a `MasterKey` on the calling thread. On many devices this is a blocking KeyStore operation that must not run on the main thread. When the user touched the Wallet tab, this constructor ran on the UI thread and either threw a `IllegalStateException`/keystore error or caused the system to kill the app, producing the "wallet falls" symptom.

## Fix

- Changed `WalletManager` to accept an already-created `IdentityManager` instead of constructing one (and therefore `IdentityStore`) in its own constructor.
- Changed `WalletScreen` to pass the existing `IdentityManager` it already receives from `MainShell` directly into `WalletManager`.
- Updated `ServiceLocator.walletManager(...)` and `ServiceLocator.thirdwebContractService(...)` to use the pre-injected `IdentityManager`.
- Added a crash guard inside `WalletScreen` so even if `loadAssets()` fails unexpectedly, the wallet tab displays an error card instead of crashing the app.
- Cleaned up several unused-parameter compiler warnings.
